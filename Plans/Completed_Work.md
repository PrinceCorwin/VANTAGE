### May 7, 2026 (Progress Grid — Bulk Row Selection Refactor: Ctrl+A No Longer Freezes at Scale)
- **Root cause confirmed.** `sfActivities` is configured `SelectionMode="Extended" SelectionUnit="Any"` so cell-level features (multi-cell copy/paste, Count/Sum/Avg stats panel) keep working. Under that configuration, `SfDataGrid.SelectAll()` materializes one `GridCellInfo` per (row × column) — at 36k rows × ~30 columns ≈ 1M `GridCellInfo` objects allocated synchronously on the UI thread inside Syncfusion's own code. Every previously-known Select-All path (Ctrl+A, Shift+Click first→last, `Actions → Select All` menu) hit this and froze. Detaching `SelectionChanged` around the call only silenced our handler, not the underlying allocation.
- **Refactor: bulk selection no longer engages Syncfusion's cell-selection model at all.** New transient `Activity.IsBulkSelected` (`INotifyPropertyChanged`, not persisted, not synced — pure UI state on the in-memory object). `Styles/RecordOwnershipRowStyleSelector.cs` extended with a `DataTrigger` bound to that flag that sets `VirtualizingCellsControl.Background` to whatever brush the host view supplies via the new `BulkSelectionBrush` property (or `AccentColor` as fallback). `Views/ProgressView.xaml.cs.SyncBulkSelectionBrushFromGrid()` copies `sfActivities.RowSelectionBrush` into the selector on view load AND on `OnThemeChanged`, so the bulk-select highlight always matches Syncfusion's native mouse-row-select color.
- **`SelectAllFilteredRowsAsync` rewritten.** Iterates `sfActivities.View?.Records` (so it respects every active filter) in 500-row chunks with `Task.Delay(1)` between chunks for UI responsiveness, flips `IsBulkSelected = true` on each row, tracks the set in a private `_bulkSelectedRows` list, and populates `sfActivities.SelectedItems` so existing bulk action handlers (`MenuAssignToUser_Click`, `MenuDuplicateRows_Click`, `MenuExportSelected_Click`, etc.) keep reading the same set. **Does not call `sfActivities.SelectAll()` anywhere.** Zero `GridCellInfo` allocation, no UI freeze at 100k-row scale.
- **`ClearBulkSelectionInternal`** — unflips just the rows we tracked in `_bulkSelectedRows` (no full-grid scan). Called automatically at the top of `sfActivities_SelectionChanged` so any fresh cell-click resets the bulk state cleanly.
- **Ctrl+A intercepted at the UserControl level.** Added to `UserControl_PreviewKeyDown` alongside the existing Ctrl+Z/Y/V handlers. Earlier attempt placed the intercept on `sfActivities.PreviewKeyDown` — that worked when no row was focused but failed when a cell had focus because the preview event tunneling order let Syncfusion's internal handler fire first at the deeper level. Putting it on the outermost level fires before any cell-level handler regardless of focus depth.
- **`RunSelectAllWithOverlayAsync` extracted** so the menu and Ctrl+A share the loading-overlay flow without duplicated code. `MenuSelectAll_Click` now just `await`s the helper before reopening the Actions context menu.
- **Mouse-driven Shift+Click first→last row at scale still freezes.** That's Syncfusion's own range-extend gesture allocating the same `GridCellInfo` rectangle, and we can't intercept it cleanly from outside the grid. Documented as a known limitation; user said "deal with it another time, if ever." Ctrl+A is the supported scale-safe path.
- **Earlier-in-session attempts that did not work** (recorded so we don't try them again): (1) Replacing the LINQ `GroupBy + Distinct + ToList` in `sfActivities_SelectionChanged` with a HashSet pass and adding a `LargeSelectionCellThreshold = 1000` guard around `UpdateSelectionStats` — this only helped if the handler was running, but the handler was already detached during `SelectAll()`; the underlying `GridCellInfo` allocation in Syncfusion was the actual blocker. (2) Removing `sfActivities.SelectAll()` from the menu path entirely without a replacement visual — `SelectedItems` alone does not drive row paint under `SelectionUnit="Any"`, so users got correct selection state with no visible highlight. (3) Hardcoding the bulk-select highlight to `AccentColor` — works but doesn't match the Syncfusion mouse-select color. (4) Placing the Ctrl+A intercept on `sfActivities.PreviewKeyDown` — fired only when no cell had focus.
- **Build clean** — 0 errors. Pre-existing `ManageSnapshotsDialog.xaml.cs:597` nullable-tuple warning unchanged.
- **Key files:** `Models/Activity.cs` (new `IsBulkSelected` transient INPC property in a new "Transient View State" section above the INPC implementation block), `Styles/RecordOwnershipRowStyleSelector.cs` (`BulkSelectionBrush` property + `DataTrigger` on `IsBulkSelected`), `Views/ProgressView.xaml.cs` (Ctrl+A in `UserControl_PreviewKeyDown`, `RunSelectAllWithOverlayAsync` helper, `SelectAllFilteredRowsAsync` rewritten, `ClearBulkSelectionInternal` + `_bulkSelectedRows` field, `SyncBulkSelectionBrushFromGrid` called from `OnViewLoaded` and `OnThemeChanged`, `sfActivities_SelectionChanged` clears bulk-select on cell interaction), `Plans/Decisions.md` (new UI/UX entry documenting the architecture choice).
