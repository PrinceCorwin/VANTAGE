### April 20, 2026 (ProgressLog UserID Captures Uploader + Original AssignedTo)
- **Root cause identified.** The admin "Upload to Progress Log" flow was losing the original record owner. `UserID` in `VANTAGE_global_ProgressLog` was hardcoded to the admin who ran the upload (`AdminSnapshotsDialog.UploadSnapshotsToProgressLog` line 399-401), and the two rows that were supposed to wire `AssignedTo` through `VMS_ColumnMappings` were both broken (one row had `ColumnName=AssignedTo` with no `AzureName`, another had `AzureName=UserID` with no `ColumnName` — the mapping loader requires both fields non-empty so neither row did anything). - **Fix: concatenate uploader and original AssignedTo into `UserID`.** Format is `"uploader|assignedto"` (e.g. `"steve|Grant.Gilbert"`). Pipe separator chosen because it can't appear in a Windows username, so parsing back to two parts is unambiguous. Concatenation happens server-side in the existing `INSERT ... SELECT FROM VMS_ProgressSnapshots` — zero extra queries, zero extra I/O, nanoseconds per row of CPU for the string concat. `[AssignedTo]` was already in the query plan's working set via the WHERE clause, so no new column reads.
- **Safety wrapper: `LEFT(CAST(... AS NVARCHAR(MAX)), maxLen)` using the `columnMaxLengths` lookup already loaded from `INFORMATION_SCHEMA`.** Prevents the whole INSERT from failing with SQL Server error 8152 (`String or binary data would be truncated`) if the combined string ever exceeds the `UserID` column's max length (currently char(50)). Cost: essentially zero — two CPU-only ops per row, invisible against the INSERT's own I/O and trigger work.
- **Backwards compatibility:** Legacy ProgressLog rows still have `UserID = 'steve'` style values (just the uploader, no pipe). Reporting queries that use `LIKE '%|Grant.Gilbert'` or `LIKE 'steve|%'` naturally skip legacy rows, which is usually the desired behavior. Legacy rows remain unfixable without the original snapshots, most of which have been purged past the 21-day retention window.
- **Decision: the concat is the permanent fix.** No column will be added to `VANTAGE_global_ProgressLog`. The pipe-separated `UserID` (`"uploader|assignedto"`) is the final shape. Backlog item removed from `Plans/Project_Status.md`.
- **Fix still pending in `VMS_ColumnMappings`:** The two broken rows (`ColumnName=AssignedTo` with null AzureName, and orphan `AzureName=UserID` with null ColumnName) can be deleted at the admin's discretion — they're inert today and don't affect the new concat behavior. Leaving them in place is harmless; they just clutter the mappings table.

### April 18, 2026 (Edit Rules Overhaul — Validator, Filter Stickiness, Export Dialog Theming)
- **New `Utilities/ActivityValidator.cs`:** Single source of truth for Activity date/percent hard rules — future dates, ActFin before ActStart, ActStart with %=0, ActFin with %&lt;100. - **Hard rules vs. required metadata:** Removed the "ActStart required when %&gt;0" and "ActFin required when %=100" checks from the validator. These are now treated as required metadata only — the cell turns red and sync is blocked, but the edit/paste succeeds. Reason: the old hard blocks created deadlocks when raising records forward without dates in hand, and users explicitly wanted those to turn into visual flags instead of blockers.
- **Single-cell ActStart/ActFin rollbacks revert to original, no auto-correction:** Previously, entering a future date clamped to today, entering ActStart&gt;ActFin silently bumped ActFin, etc. Now every hard-rule violation reverts the cell to its pre-edit value and shows the validation message. Refactored the two blocks into one shared path that calls the validator.
- **Entering ActFin auto-bumps % to 100:** On single-cell edit, Find &amp; Replace, and paste, a non-null ActFin value on a row where % &lt; 100 now sets % to 100 before validation.
