### May 19, 2026 (SkySkraper — Synology Drive On-Demand-Sync Zombie Cleanup)

**Day-after fallout from the path-shortening renames.** Duplicate folders appeared under SkySkraper's `output/workbooks/` and `output/cdx_workbooks/` — both the long-form folder and its new short-form twin existed as siblings.

**Root cause:** the work PC's Synology Drive client was running in **on-demand sync** mode during the rename pass. On-demand sync stores files as cloud placeholders, not real local files. When the rename script issued thousands of `rename(src, dst)` calls in rapid succession, SDrive raced with itself — it interpreted each rename as `delete src` + `create dst`, then "repaired" the missing `src` by re-downloading it from the NAS. Result: every renamed folder had its old long-form copy resurrected from the cloud. Home Legion PCs were powered off during the rename window, so they were ruled out as the source.

**Diagnosed clean:** a diagnostic script classified every long-form path on disk as a *zombie* (long-form whose fully-shortened twin also exists) — 0 orphans, meaning the rename script didn't miss anything; every long path had a matching short twin. A 20-pair sample compared byte-identical.

**Cleanup performed on the work PC:** switched the SkySkraper SDrive task to "Sync everything to this computer", waited for the tree to materialize locally, paused sync, ran the cleanup script with safety checks (refused to delete anything without a verified short-form twin). Deleted **195 top-level zombie paths** (67 in `workbooks/`, 128 in `cdx_workbooks/`), 0 refused. Re-ran the diagnostic to confirm 0 zombies / 0 orphans.

**Re-cleanup on the personal Legion (later same day).** Steve powered on the old Legion (the second home PC) — it had been off during both the rename and the work-PC cleanup, so its SDrive client was still holding the original pre-rename long-form tree. On startup it reconciled with the NAS, interpreted the work-PC deletions as "missing files" on its end, and pushed them back up. Those resurrected paths then synced down to the new personal Legion. Steve uninstalled SDrive from the old Legion to close that loophole permanently, then re-ran the same cleanup procedure on the new Legion: SDrive already on full-sync + paused, diagnostic showed **190 zombie dirs + 31 zombie files + 8 orphan files**, `delete_zombies.py --apply` removed 100 top-level paths (covering all 221 zombies; descendants rmtree'd with parents), and re-running `shorten_workbook_paths.py --apply` caught the 8 orphans — all in `Slip Coupling Shrink Sleeve` filenames where the original substitution table's `Slip-Coupling-w-Shrink-Sleeve → Slip-Coupling-Shrink-Sleeve` rule did exist but didn't fire on these particular files in the original pass (likely a transient I/O ordering issue, since the rule is correct and idempotent — the re-run renamed them cleanly with 0 conflicts). Final diagnostic: 0/0 across both `workbooks/` and `cdx_workbooks/`. NAS-side recycle bin still empty at this point because the new Legion's sync was still paused; deletions will propagate after Steve resumes sync, at which point the NAS bin must be emptied again before any other PC reconnects.

**Operational rule going forward (applies to any SDrive-synced folder used by SkySkraper or VANTAGE — e.g., the AI Takeoff `Conversion/` folder):** before any bulk rename/delete inside an SDrive-synced folder, set the sync task to "Sync everything to this computer" and wait for the tree to be fully local (optionally pause sync during the op). After the operation, before resuming sync on other machines, empty the NAS Synology Drive Recycle Bin for the affected shared folder — long-form files there can resurrect. **And** account for any PC that was offline during the original op: when it comes back online its SDrive client will treat the new short-form tree as "missing files" and try to restore them from its stale local copy. Either pre-empt by uninstalling/disabling SDrive on that PC before it boots, or be ready to repeat the cleanup once it reconciles.

**Authoritative narrative + full diagnostic detail:** `SkySkraper/SynologyDrive/Plans/cdx_Project_Status.md`, May 19 section.

**Key files:** SkySkraper-side only (no VANTAGE code touched). Throwaway scripts marked for deletion after the pending generator patches land — see Project_Status.md.

### May 7, 2026 (Progress Grid — Bulk Row Selection Refactor: Ctrl+A No Longer Freezes at Scale)
- **Root cause confirmed.** `sfActivities` is configured `SelectionMode="Extended" SelectionUnit="Any"` so cell-level features (multi-cell copy/paste, Count/Sum/Avg stats panel) keep working. Under that configuration, `SfDataGrid.SelectAll()` materializes one `GridCellInfo` per (row × column) — at 36k rows × ~30 columns ≈ 1M `GridCellInfo` objects allocated synchronously on the UI thread inside Syncfusion's own code. Every previously-known Select-All path (Ctrl+A, Shift+Click first→last, `Actions → Select All` menu) hit this and froze. Detaching `SelectionChanged` around the call only silenced our handler, not the underlying allocation.
- **Refactor: bulk selection no longer engages Syncfusion's cell-selection model at all.** New transient `Activity.IsBulkSelected` (`INotifyPropertyChanged`, not persisted, not synced — pure UI state on the in-memory object). `Styles/RecordOwnershipRowStyleSelector.cs` extended with a `DataTrigger` bound to that flag that sets `VirtualizingCellsControl.Background` to whatever brush the host view supplies via the new `BulkSelectionBrush` property (or `AccentColor` as fallback). `Views/ProgressView.xaml.cs.SyncBulkSelectionBrushFromGrid()` copies `sfActivities.RowSelectionBrush` into the selector on view load AND on `OnThemeChanged`, so the bulk-select highlight always matches Syncfusion's native mouse-row-select color.
- **`SelectAllFilteredRowsAsync` rewritten.** Iterates `sfActivities.View?.Records` (so it respects every active filter) in 500-row chunks with `Task.Delay(1)` between chunks for UI responsiveness, flips `IsBulkSelected = true` on each row, tracks the set in a private `_bulkSelectedRows` list, and populates `sfActivities.SelectedItems` so existing bulk action handlers (`MenuAssignToUser_Click`, `MenuDuplicateRows_Click`, `MenuExportSelected_Click`, etc.) keep reading the same set. **Does not call `sfActivities.SelectAll()` anywhere.** Zero `GridCellInfo` allocation, no UI freeze at 100k-row scale.
- **`ClearBulkSelectionInternal`** — unflips just the rows we tracked in `_bulkSelectedRows` (no full-grid scan). Called automatically at the top of `sfActivities_SelectionChanged` so any fresh cell-click resets the bulk state cleanly.
- **Ctrl+A intercepted at the UserControl level.** Added to `UserControl_PreviewKeyDown` alongside the existing Ctrl+Z/Y/V handlers.
