### May 5, 2026 (FS Multiplier Fix + Branch/Toggle Pivot + Takeoff Rate Mode PRD)
- **`Services/AI/TakeoffPostProcessor.cs:ApplyRates` — FS labor rows no longer double-multiplied.** Field Support (FS) rates in `Resources/RateSheet.json` already have `RollupMult` and material-group multipliers baked into the published rate value. The rate-application pass was multiplying them again, inflating every FS row's `BudgetMHs`. Fix: when `component == "FS"`, force both factors to 1.0 in the audit columns AND in the BudgetMHs computation (`BudgetMHs = mhu × 1 × 1 × qty`). Other components unaffected. Audit columns now honestly show 1.0 for FS rows so the user can verify at a glance that no multiplier was applied. Discovered by user during MCAA parity-test scoping; bug was longstanding under Summit pricing.
- **Brief branching exercise then reversal.** Created `feature/mcaa-takeoff` branch from commit `907b591` (TUBE footage handling + CUT companion rows on BW/SW) to isolate in-progress MCAA-prep takeoff changes from the publishable main. Cherry-picked the FS fix onto the branch so both lines stayed in sync. Added `feature/mcaa-takeoff`-only commit skipping BOLT/GSKT/WAS labor-row creation (MCAA captures bolt-up labor in the joint rate, not as separate hardware lines; under Summit, those rows still get priced and stay).
- **Doc-on-main-only `CLAUDE.md` rule briefly tried, then walked back via `git revert`.** Added a Workflow Skills sub-section saying all `Plans/*.md`, `Help/manual.html`, `README.md`, and `CLAUDE.md` updates happen on main only — never on feature branches.
