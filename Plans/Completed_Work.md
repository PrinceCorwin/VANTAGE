### June 25, 2026 (Hotfix â€” Reassign SyncVersion Regression + DeletedRecords Refresh/Timeout + Adaptive Pull Plan)

**Reassign UPDATE no longer writes SyncVersion explicitly.** `Views/ProgressView.xaml.cs:6133-6144` â€” the bulk reassign UPDATE used to include `SyncVersion = SyncVersion + 1` in its SET clause. Pre-2026-06-24 this was harmless theater because `TR_VMS_Activities_SyncVersion` fired unconditionally and overwrote the `+1` with a fresh value from `VMS_GlobalSyncVersion`. Yesterday's conditional-trigger change (`IF UPDATE(SyncVersion) RETURN`) removed that safety net â€” the explicit write now short-circuits the trigger and the per-row `+1` becomes the actual final SyncVersion, far below every user's `LastPulledSyncVersion`. Symptom (reported live): user reassigns a batch of records; popup says "Assignment worked"; email fires correctly; the assigner's grid never updates because the post-reassign `PullRecordsAsync` returns zero matching rows (`WHERE SyncVersion > lastPulled` excludes the `+1`-bumped rows); the recipient never sees the rows on their next sync either. Audit found ONE writer with this latent bug â€” the reassign UPDATE â€” every other Azure UPDATE on `VMS_Activities` (push staging, soft-delete, ProgressLog AzureUploadUtcDate update) correctly omits SyncVersion from its SET clause and lets the trigger handle it. Fix: removed the `SyncVersion = SyncVersion + 1` clause. Inline comment added to call out why the omission is load-bearing so future-us doesn't add it back.

**DeletedRecordsView project-list query no longer times out.** `Views/DeletedRecordsView.xaml.cs:81-83` â€” `SELECT DISTINCT ProjectID FROM VMS_Activities WHERE IsDeleted = 1` was running on the default 30-second SqlCommand timeout. With no index covering `IsDeleted`, the query degenerated to a clustered scan of the 15GB+ table; under today's elevated Azure load it consistently exceeded the timeout and threw `Execution Timeout Expired` when admins tried to open the Deleted Records dialog. Added `CommandTimeout = 300` to match the sibling per-project query at line 169-170. Immediate unblock for admins on the deployed binary once the hotfix ships.

**Azure filtered index for deleted-records queries.** `DatabaseSetup.cs` â†’ `EnsureAzureIndexes` now creates `IX_VMS_Activities_Deleted_ProjectID ON VMS_Activities (ProjectID) WHERE IsDeleted = 1`. Filtered index is tiny (only deleted rows participate) and exactly serves both the DISTINCT-projects load and the per-project Refresh query. Auto-creates on next app launch whenever a user with CREATE INDEX permission opens the app. Equivalent SSMS statement (`WITH (ONLINE = ON)` so syncs aren't blocked) handed to the admin so they could unblock the bug immediately without waiting for the hotfix.

**DeletedRecordsView grid refresh now awaits the load instead of fire-and-forgetting.** `Views/DeletedRecordsView.xaml.cs` â€” `BtnRefresh_Click` was `async void`; `BtnPurge_Click` (line 619) and `BtnRestore_Click` (line 522) called it directly without await, so the calls returned at the first `await Task.Run(...)` inside the refresh. The calling handler's `finally` block then ran (`SetActionBusy(false)`, `txtStatus.Text = "Ready"`), making it look like the refresh was done while the Azure query was still pending. By the time the refresh actually completed, the user had already given up and clicked Refresh manually. Refactor: extracted the load body into `private async Task LoadDeletedRecordsAsync()`, made `BtnRefresh_Click` a one-line wrapper that awaits it, and swapped both Purge and Restore handlers to `await LoadDeletedRecordsAsync()`. Busy indicator now stays on through the full refresh cycle and the grid updates correctly the moment Azure responds. Refresh logger key renamed to `"DeletedRecordsView.LoadDeletedRecordsAsync"` for accuracy.

**Pull adapts query plan to expected row count.** `Utilities/SyncManager.cs` â†’ `PullRecordsAsync` now runs a cheap `SELECT COUNT(*)` against the existing `IX_VMS_Activities_Project_SyncVersion` index before the main pull. If the count exceeds `ForceScanThreshold` (5,000), the SELECT is built with `WITH (INDEX(0))` to force a clustered scan, and `ORDER BY [SyncVersion]` is dropped (would force a sort on top of the scan and spill to tempdb at 100k+ rows). Below the threshold, the SELECT keeps the index-seek path and the ORDER BY. Backstory: the index added 2026-06-24 was non-covering â€” small-delta pulls got the advertised 31s â†’ 1.4s speedup on single-row pulls, but `SELECT *` over thousands of rows forced thousands of key lookups back to the clustered index and was 2.6Ă— slower than a clustered scan at scale. SSMS diagnostic (`SET STATISTICS IO/TIME ON`) confirmed: 125k-row pull = 622k logical reads + 107s on the index plan vs 147k reads + 41s with `WITH (INDEX(0))`. `UPDATE STATISTICS WITH FULLSCAN` did not move the optimizer's plan choice. Field test post-fix: 39k-row pull dropped from ~110s â†’ ~30s. Log line `Project X: expecting N rows, plan=clustered scan|index seek` ships so we can verify the picker in production.

**Project_Status.md Critical section.** Dropped the "Existing-drift cleanup from past trigger-disable windows" bullet â€” analysis showed the drift is self-revealing: any user with stale ownership on a row gets the "No longer assigned to you" push reject the moment they edit + sync it, no need for a one-shot SyncVersion bump that would have its own operational risk. Rewrote the "Push re-claim of reassigned rows" bullet to reflect the actual code paths: `SyncManager.cs:281-296` already gates push UPDATEs on ownership, so the common-case "edit at 9am / admin reassigns at 10am / user syncs at 11am" sequence is correctly rejected with "No longer assigned to you" â€” no silent overwrite. Real bug is a narrow Step 3 race within a single push call (between the ownership SELECT at `:267-279` and the staging UPDATE at `:354-363`): admin reassigns landing in that ~1â€“3s window get silently overwritten. Listed fix shapes: (a) wrap Steps 2/3 in a SERIALIZABLE transaction (lock-pile-up bounded â€” disjoint UniqueIDs per user) or (b) compare-and-set via `OriginalAssignedTo` on the staging table.

### June 3, 2026 (AI Takeoff â€” REDT and REDC Generate the Correct Number of Weld Labor Rows)

**Problem.** `TakeoffPostProcessor.BuildConnectionPairs` had a dedicated dual-size branch for `TEE` that correctly emitted 3 connection pairs (2 run faces + 1 outlet) but routed every other dual-size component through a generic "1 larger + 1 smaller" branch. That undercounted REDT (a reducing tee â€” same topology as TEE, 3 welds) by one row and REDC (a reducing cross â€” 4 welds) by two rows. For example, REDT `1x0.5` with `Connection Qty=3, Connection Type=BW` was producing 1Ă—BW@1 + 1Ă—BW@0.5 (2 labor rows) instead of 2Ă—BW@1 + 1Ă—BW@0.5 (3 rows). `Connection Qty` from the BOM was being ignored by that branch entirely.

**Fix in `TakeoffPostProcessor.BuildConnectionPairs`.** Folded REDT into the existing TEE dual-size branch (`(component == "TEE" || component == "REDT") && isDualSize`) so both emit 3 pairs: `runTypeĂ—largerSize`, `runTypeĂ—largerSize`, `outletTypeĂ—smallerSize`. Added a new REDC dual-size branch immediately below that emits 4 pairs: 2Ă— run at larger + 2Ă— outlet at smaller. RED / SWG / REDE are genuine 2-weld reducers, so they still flow through the generic dual-size branch unchanged.

**Fix in `FittingMakeupService.CalculateFittingMakeupForPipe`.** Added a `REDC` branch that mirrors the existing REDT pattern: parses the `4x2`-style size, routes same-size REDC to `CROSS` (4Ă— run makeup) as a safety net, and for true reducing crosses contributes `RunIn * 2` when the calling pipe is the larger run size and `OutletIn * 2` when the calling pipe is the smaller branch size. Without this branch, REDC was hitting the catch-all "single weld" path and contributing only 1Ă— makeup regardless of topology.

**Known data gap.** `Resources/FittingMakeup.json` currently has zero REDC entries (vs. 285 REDT entries). The labor-row count fix above works standalone, but REDC makeup-inches lookups will now land on the Missed Makeups tab with key `BW/REDC/{larger}x{smaller}/...` until rate-sheet entries are added. That's intentionally surfacing the gap rather than the prior behavior of silently treating REDC as a 1-weld fitting.

**Key files:** `Services/AI/TakeoffPostProcessor.cs` (BuildConnectionPairs â€” REDT/REDC branches), `Services/AI/FittingMakeupService.cs` (CalculateFittingMakeupForPipe â€” REDC handler).
